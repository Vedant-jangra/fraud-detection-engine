1.2 Simulating Realistic Imbalanced Fraud Data
Real fraud datasets are confidential. We simulate one with statistically correct properties: 0.1%–0.5%
fraud rate (severe class imbalance), realistic transaction amounts following a log-normal distribution,
temporal patterns (weekday vs weekend), and distinct fraud archetypes (card testing, account
takeover, bust-out fraud).

import numpy as np
import pandas as pd
from faker import Faker
from datetime import datetime, timedelta
import json, uuid, random
fake = Faker('en_IN')  # Indian locale for realistic data
rng  = np.random.default_rng(seed=42)  # Reproducible
FRAUD_ARCHETYPES = {
    'card_testing':      {'amount_range': (1, 50),    'velocity': 'high', 'weight': 0.35},
    'account_takeover':  {'amount_range': (5000, 50000), 'velocity': 'med', 'weight': 0.40},
    'bust_out':          {'amount_range': (1000, 10000), 'velocity': 'low', 'weight': 0.25},
}
def generate_transaction(user_id: str, is_fraud: bool = False) -> dict:
    base_amount = float(rng.lognormal(mean=6.5, sigma=1.2))  # Log-normal dist
    if is_fraud:
        archetype = random.choices(
            list(FRAUD_ARCHETYPES.keys()),
            weights=[v['weight'] for v in FRAUD_ARCHETYPES.values()]
        )[0]
        lo, hi  = FRAUD_ARCHETYPES[archetype]['amount_range']
        amount  = round(random.uniform(lo, hi), 2)
    else:
        amount  = round(base_amount, 2)
    return {
        'transaction_id':  str(uuid.uuid4()),
        'user_id':         user_id,
        'timestamp':       datetime.utcnow().isoformat() + 'Z',
        'amount':          amount,
        'merchant_id':     fake.bothify(text='MER-####??'),
        'merchant_cat':    random.choice(['grocery','travel','electronics',
                                          'restaurant','utility','crypto']),
        'country':         random.choices(['IN','US','SG','AE'],
                                          weights=[0.70,0.15,0.10,0.05])[0],
        'device_id':       fake.md5()[:16],
        'ip_address':      fake.ipv4(),
        'is_fraud':        int(is_fraud),   # Label (hidden at inference time)
    }
def generate_dataset(n_transactions: int = 100_000,
                     fraud_rate: float = 0.003) -> pd.DataFrame:
    user_pool   = [fake.uuid4() for _ in range(10_000)]
    fraud_count = int(n_transactions * fraud_rate)
    legit_count = n_transactions - fraud_count
    records = (
        [generate_transaction(random.choice(user_pool), False) for _ in range(legit_count)]
        + [generate_transaction(random.choice(user_pool), True)  for _ in range(fraud_count)]
    )
    df = pd.DataFrame(records).sample(frac=1, random_state=42).reset_index(drop=True)
    print(f'Generated {len(df):,} transactions | Fraud rate: {df.is_fraud.mean():.3%}')
    return df

TimescaleDB extends PostgreSQL with hypertables — automatically partitioned by time. This makes
time-range queries on billions of rows 10–100x faster than standard PostgreSQL. Understanding this
is critical for velocity feature engineering in Phase 2.
SQL
-- 001_init_schema.sql
-- Run: psql -h localhost -U postgres -d fraud_db -f 001_init_schema.sql
CREATE EXTENSION IF NOT EXISTS timescaledb;
-- Core transaction table (will become a hypertable)
CREATE TABLE transactions (
    transaction_id   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          TEXT        NOT NULL,
    timestamp        TIMESTAMPTZ NOT NULL,
    amount           NUMERIC(12, 2) NOT NULL,
    merchant_id      TEXT,
    merchant_cat     TEXT,
    country          CHAR(2),
    device_id        TEXT,
    ip_address       INET,
    is_fraud         SMALLINT    DEFAULT 0
);
-- Convert to hypertable partitioned by hour
SELECT create_hypertable('transactions', 'timestamp',
                         chunk_time_interval => INTERVAL '1 hour');
-- Continuous aggregate for hourly velocity stats
CREATE MATERIALIZED VIEW user_hourly_stats
WITH (timescaledb.continuous) AS
    SELECT
        user_id,
        time_bucket('1 hour', timestamp)   AS bucket,
        COUNT(*)                            AS txn_count,
        SUM(amount)                         AS total_amount,
        AVG(amount)                         AS avg_amount
    FROM transactions
    GROUP BY user_id, bucket
WITH NO DATA;
-- Auto-refresh every 30 seconds
SELECT add_continuous_aggregate_policy('user_hourly_stats',
    start_offset  => INTERVAL '2 hours',
    end_offset    => INTERVAL '30 seconds',
    schedule_interval => INTERVAL '30 seconds');
-- Indexes for fast feature lookups
CREATE INDEX idx_txn_user_time ON transactions (user_id, timestamp DESC);
CREATE INDEX idx_txn_device    ON transactions (device_id, timestamp DESC);
CREATE INDEX idx_txn_ip        ON transactions (ip_address, timestamp DESC);
1.4 Architecture Design Decisions
Every architectural choice involves trade-offs. Interviewers at ■30 LPA levels will probe these. Here
is the rationale for this system's key decisions:
■ Why Kafka over direct API calls?
Kafka decouples producers from consumers, provides replay capability (critical for debugging fraud
cases), and handles traffic spikes via buffering. At 10K TPS, synchronous API calls would cascade
failures.
■ Why TimescaleDB over Redis for the Feature Store?

---PAGE 9---

Real-Time High-Frequency Fraud & Financial Intelligence Engine    12-Week Mastery Roadmap
Page 9
Redis is faster but loses data on restart (without persistence config). TimescaleDB gives us SQL
expressiveness for complex window queries, ACID guarantees, and continuous aggregates that
auto-compute hourly stats. For < 10ms feature lookups, both work; TimescaleDB wins on query
complexity.
■ Why ONNX over serving the native XGBoost model?
ONNX Runtime applies hardware-specific optimizations (SIMD, graph fusion) automatically. In
benchmarks, ONNX-quantized XGBoost is 2–4x faster than native Python inference. This is the
difference between 40ms and 18ms p99 latency.
■ Why FastAPI over Flask?
FastAPI is ASGI-native, meaning it handles async I/O without blocking. One sync endpoint in Flask
under heavy load blocks the entire process. FastAPI's async handlers allow concurrent feature store
queries during inference.

# ingestion/kafka_producer.py
from confluent_kafka import Producer
from data.generators.transaction_simulator import generate_transaction
import json, time, random, logging
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)
KAFKA_CONFIG = {
    'bootstrap.servers': 'localhost:9092',
    'client.id':         'fraud-producer',
    # Performance tuning — critical for high-throughput
    'linger.ms':         5,       # Batch records for 5ms before sending
    'batch.size':        16384,   # 16KB batches
    'compression.type':  'snappy',# ~40% size reduction
    'acks':              '1',     # Leader ack only (throughput vs durability trade-off)
}
def delivery_report(err, msg):
    if err:
        logger.error(f'Message delivery failed: {err}')
def stream_transactions(tps: int = 1000, duration_seconds: int = 300):
    producer  = Producer(KAFKA_CONFIG)
    user_pool = [f'user_{i:05d}' for i in range(10_000)]
    interval  = 1.0 / tps
    start     = time.monotonic()
    count     = 0
    while time.monotonic() - start < duration_seconds:
        is_fraud = random.random() < 0.003  # 0.3% fraud rate
        txn      = generate_transaction(random.choice(user_pool), is_fraud)
        producer.produce(
            topic     = 'transactions',
            key       = txn['user_id'].encode(),  # Key = user_id for partitioning
            value     = json.dumps(txn).encode(),
            callback  = delivery_report
        )
        producer.poll(0)  # Non-blocking delivery callback check
        count += 1
        time.sleep(interval)
    producer.flush()  # Ensure all buffered messages are sent
    logger.info(f'Streamed {count:,} transactions in {duration_seconds}s')
if __name__ == '__main__':
    stream_transactions(tps=500, duration_seconds=600)

-- velocity_queries.sql  (Part 1/2)
-- PURPOSE: Compute velocity features comparing current vs historical behavior.
WITH
-- ■■ CTE 1: Recent Activity (last 10 minutes) ■■■■■■■■■■■■■■■■■■■■■■■■
recent_activity AS (
    SELECT
        user_id,
        COUNT(*)                          AS txn_count_10m,
        SUM(amount)                       AS total_amount_10m,
        AVG(amount)                       AS avg_amount_10m,
        MAX(amount)                       AS max_amount_10m,
        COUNT(DISTINCT merchant_id)       AS unique_merchants_10m,
        COUNT(DISTINCT country)           AS unique_countries_10m,
        COUNT(DISTINCT device_id)         AS unique_devices_10m
    FROM transactions
    WHERE
        user_id   = $1   -- parameterized, injected at query time
        AND timestamp >= NOW() - INTERVAL '10 minutes'
    GROUP BY user_id
),
-- ■■ CTE 2: Historical Baseline (rolling 30-day, excluding last hour) ■
historical_baseline AS (
    SELECT
        user_id,
        AVG(daily_count)                  AS avg_daily_txn_count,
        STDDEV(daily_count)               AS std_daily_txn_count,
        AVG(daily_amount)                 AS avg_daily_amount,
        PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY daily_amount)
                                          AS p95_daily_amount
    FROM (
        SELECT
            user_id,
            DATE_TRUNC('day', timestamp)  AS day,
            COUNT(*)                      AS daily_count,
            SUM(amount)                   AS daily_amount
        FROM transactions
        WHERE
            user_id   = $1
            AND timestamp BETWEEN NOW() - INTERVAL '30 days'
                                AND NOW() - INTERVAL '1 hour'
        GROUP BY user_id, day
    ) daily_stats
    GROUP BY user_id
),

-- ■■ CTE 3: Sliding Window Rank ■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■
window_velocity AS (
    SELECT
        user_id, timestamp, amount,
        COUNT(*) OVER (
            PARTITION BY user_id ORDER BY timestamp
            RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW
        )                                    AS txn_count_1h,
        COUNT(*) OVER (
            PARTITION BY user_id ORDER BY timestamp
            RANGE BETWEEN INTERVAL '24 hours' PRECEDING AND CURRENT ROW
        )                                    AS txn_count_24h,
        -- Z-score: how anomalous is this transaction amount for this user?
        (amount - AVG(amount) OVER (PARTITION BY user_id)) /
            NULLIF(STDDEV(amount) OVER (PARTITION BY user_id), 0)
                                             AS amount_zscore,
        -- Inter-transaction time (low gap = high velocity attack signal)
        timestamp - LAG(timestamp) OVER (
            PARTITION BY user_id ORDER BY timestamp
        )                                    AS time_since_last_txn
    FROM transactions
    WHERE user_id = $1 AND timestamp >= NOW() - INTERVAL '25 hours'
),
-- ■■ CTE 4: Velocity Ratio Calculation ■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■■
velocity_ratio AS (
    SELECT
        r.user_id,
        r.txn_count_10m,  r.total_amount_10m,
        r.unique_merchants_10m, r.unique_countries_10m,
        h.avg_daily_txn_count,  h.avg_daily_amount,
        -- KEY SIGNAL: 10-min rate vs daily average (144 = 10-min slots per day)
        CASE WHEN COALESCE(h.avg_daily_txn_count, 0) = 0 THEN NULL
             ELSE (r.txn_count_10m * 144.0) / h.avg_daily_txn_count
        END                                  AS velocity_ratio,
        CASE WHEN COALESCE(h.avg_daily_amount, 0) = 0 THEN NULL
             ELSE r.total_amount_10m / h.avg_daily_amount
        END                                  AS amount_spike_ratio
    FROM recent_activity r
    LEFT JOIN historical_baseline h USING (user_id)
)
SELECT
    vr.*,
    (velocity_ratio > 10)::int             AS high_velocity_flag,
    (amount_spike_ratio > 5)::int           AS amount_spike_flag
FROM velocity_ratio vr;
2.4 Handling Late-Arriving Data (Critical Concept)
In distributed systems, events arrive out of order. A transaction processed at T+5 minutes might have
actually occurred at T-3 minutes (network delay, mobile app sync, retries). If you ignore late data,
your 10-minute velocity window is wrong.
■■  LATE DATA PROBLEM
Example: User makes 8 transactions between 14:00-14:10. Three of them arrive at the Kafka broker at 14:14 due to
a flaky mobile connection. Without watermarking, your 10-minute window at 14:10 only sees 5 transactions and
misses the fraud signal. Watermarks define how late data can arrive before being dropped.

def engineer_features(df: pd.DataFrame) -> pd.DataFrame:
    df = df.copy()
    df['timestamp'] = pd.to_datetime(df['timestamp'])
    # Temporal features
    df['hour_of_day']     = df['timestamp'].dt.hour
    df['day_of_week']     = df['timestamp'].dt.dayofweek
    df['is_weekend']      = (df['day_of_week'] >= 5).astype(int)
    df['is_night']        = df['hour_of_day'].between(22, 6).astype(int)
    # Amount features
    df['log_amount']      = np.log1p(df['amount'])
    df['amount_rounded']  = (df['amount'] % 1 == 0).astype(int)  # Round amounts = card testing
    df['is_micro_txn']    = (df['amount'] < 10).astype(int)
    # Encode categoricals
    le = LabelEncoder()
    df['merchant_cat_enc'] = le.fit_transform(df['merchant_cat'].fillna('unknown'))
    df['country_enc']      = le.fit_transform(df['country'].fillna('XX'))
    # Velocity features (pre-computed via SQL, joined here)
    # These columns should already exist after SQL feature store join

                     'amount_spike_ratio', 'unique_merchants_10m',
                     'unique_countries_10m', 'txn_count_1h']
    for col in velocity_cols:
        df[col] = df.get(col, pd.Series(np.zeros(len(df))))
    return df

def train_xgboost(df: pd.DataFrame):
    df = engineer_features(df)
    X  = df[FEATURE_COLS].fillna(0)
    y  = df['is_fraud']
    fraud_count  = y.sum()
    legit_count  = len(y) - fraud_count
    pos_weight   = legit_count / fraud_count  # e.g. 333 for 0.3% rate
    # Apply SMOTE only on training fold
    smote = SMOTE(sampling_strategy=0.1, random_state=42, k_neighbors=5)
    skf = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
    oof_preds = np.zeros(len(X))
    xgb_params = {
        'objective':        'binary:logistic',
        'eval_metric':      'aucpr',         # Area under PR curve — right metric for fraud
        'scale_pos_weight': pos_weight,
        'max_depth':        6,
        'learning_rate':    0.05,
        'n_estimators':     500,
        'min_child_weight': 5,               # Prevents overfitting on fraud minority
        'subsample':        0.8,
        'colsample_bytree': 0.8,
        'tree_method':      'hist',          # Faster for large datasets
        'device':           'cpu',           # Change to 'cuda' if GPU available
        'early_stopping_rounds': 30,
        'random_state':     42,
    }
    models = []
    for fold, (train_idx, val_idx) in enumerate(skf.split(X, y)):
        X_tr, X_val = X.iloc[train_idx], X.iloc[val_idx]
        y_tr, y_val = y.iloc[train_idx], y.iloc[val_idx]
        X_tr_sm, y_tr_sm = smote.fit_resample(X_tr, y_tr)  # SMOTE on train only
        model = xgb.XGBClassifier(**xgb_params)
        model.fit(X_tr_sm, y_tr_sm,
                  eval_set=[(X_val, y_val)],
                  verbose=50)
        oof_preds[val_idx] = model.predict_proba(X_val)[:, 1]
        models.append(model)
        auprc = average_precision_score(y_val, oof_preds[val_idx])
        print(f'Fold {fold+1} AUPRC: {auprc:.4f}')
    overall_auprc = average_precision_score(y, oof_preds)
    print(f'Overall CV AUPRC: {overall_auprc:.4f}')
    return models[-1]  # Return last fold model (or ensemble)

import xgboost as xgb
import onnxmltools
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
import onnxruntime as rt
from onnxruntime.quantization import quantize_dynamic, QuantType
import numpy as np, time
N_FEATURES = 17  # Must match FEATURE_COLS length in train.py
def export_to_onnx(model: xgb.XGBClassifier, output_path: str):
    '''Export trained XGBoost model to ONNX format.'''
    initial_type = [('float_input', FloatTensorType([None, N_FEATURES]))]
    onnx_model   = convert_sklearn(model, initial_types=initial_type,
                                   target_opset=18)
    with open(output_path, 'wb') as f:
        f.write(onnx_model.SerializeToString())
    print(f'Exported ONNX model to {output_path}')
    return output_path
def quantize_model(onnx_path: str, quantized_path: str):
    '''Apply INT8 dynamic quantization for faster CPU inference.'''
    quantize_dynamic(
        model_input   = onnx_path,
        model_output  = quantized_path,
        weight_type   = QuantType.QInt8,   # INT8 weights
    )
    print(f'Quantized model saved to {quantized_path}')
def benchmark_inference(onnx_path: str, n_runs: int = 1000):
    '''Measure p50, p95, p99 latency for a single prediction.'''
    sess = rt.InferenceSession(onnx_path,
                               providers=['CPUExecutionProvider'])
    input_name  = sess.get_inputs()[0].name
    dummy_input = np.random.randn(1, N_FEATURES).astype(np.float32)
    # Warm-up runs (JIT compilation, cache warming)
    for _ in range(20):
        sess.run(None, {input_name: dummy_input})
    latencies = []
    for _ in range(n_runs):
        t0 = time.perf_counter()
        sess.run(None, {input_name: dummy_input})
        latencies.append((time.perf_counter() - t0) * 1000)  # ms
    latencies.sort()
    print(f'Inference latency (n={n_runs}):',
          f'p50={latencies[int(n_runs*0.50)]:.2f}ms',
          f'p95={latencies[int(n_runs*0.95)]:.2f}ms',
          f'p99={latencies[int(n_runs*0.99)]:.2f}ms')

@app.post('/v1/score', response_model=FraudResponse)
async def score_transaction(request: TransactionRequest):
    t_start = time.perf_counter()
    REQUESTS_TOTAL.inc()
    # Step 1: Async feature retrieval from TimescaleDB (~6-8ms)
    features = await app.state.extractor.get_features(
        user_id      = request.user_id,
        amount       = request.amount,
        merchant_cat = request.merchant_cat,
        country      = request.country,
        timestamp    = request.timestamp,
    )
    # Step 2: ONNX inference (~1-3ms after INT8 quantization)
    fraud_prob = app.state.predictor.predict(features)
    latency_ms = (time.perf_counter() - t_start) * 1000
    INFERENCE_LATENCY.observe(latency_ms)
    FRAUD_SCORE_HIST.observe(fraud_prob)
    if fraud_prob > 0.7:

    return FraudResponse(
        transaction_id    = request.transaction_id,
        fraud_probability = round(fraud_prob, 4),
        risk_level = 'HIGH'   if fraud_prob > 0.7 else
                     'MEDIUM' if fraud_prob > 0.3 else 'LOW',
        latency_ms    = round(latency_ms, 2),
        features_used = features.feature_names,
    )

# processing/drift_detector.py
import numpy as np
import pandas as pd
from prometheus_client import Gauge
import asyncpg, logging
logger = logging.getLogger(__name__)
PSI_GAUGE   = Gauge('feature_psi', 'Population Stability Index',
                    ['feature_name'])
MODEL_PERF  = Gauge('model_precision_rolling_1h', 'Rolling 1h precision')
def compute_psi(expected: np.ndarray, actual: np.ndarray,
                n_bins: int = 10) -> float:
    '''
    Population Stability Index (PSI):
    PSI < 0.1  -> No significant change (stable)
    PSI 0.1-0.2 -> Moderate drift (investigate)
    PSI > 0.2  -> Major shift (retrain model!)
    '''
    _, bin_edges   = np.histogram(expected, bins=n_bins)
    bin_edges[0]   = -np.inf
    bin_edges[-1]  =  np.inf
    expected_pct = np.histogram(expected, bins=bin_edges)[0] / len(expected)
    actual_pct   = np.histogram(actual,   bins=bin_edges)[0] / len(actual)
    # Avoid log(0) — replace zeros with small epsilon
    expected_pct = np.where(expected_pct == 0, 1e-6, expected_pct)
    actual_pct   = np.where(actual_pct   == 0, 1e-6, actual_pct)
    psi = np.sum((actual_pct - expected_pct)
                 * np.log(actual_pct / expected_pct))
    return float(psi)

async def run_drift_check(db_pool: asyncpg.Pool,
                          training_stats: dict) -> dict:
    '''
    Fetch last 1h of production data, compute PSI per feature,
    update Prometheus gauges. Called by Spark every 30 minutes.
    '''
    async with db_pool.acquire() as conn:
        rows = await conn.fetch(
            '''SELECT amount, txn_count_10m, velocity_ratio
               FROM transactions
               WHERE timestamp >= NOW() - INTERVAL '1 hour'
               LIMIT 10000'''
        )
    if not rows:
        logger.warning('No production data for drift check')
        return {}
    prod_df = pd.DataFrame(rows,
                columns=['amount', 'txn_count_10m', 'velocity_ratio'])
    drift_report = {}
    for feature in ['amount', 'txn_count_10m', 'velocity_ratio']:
        psi = compute_psi(
            expected = training_stats[feature],  # Saved at training time
            actual   = prod_df[feature].dropna().values
        )
        PSI_GAUGE.labels(feature_name=feature).set(psi)
        drift_report[feature] = psi
        if psi > 0.2:
            logger.critical(
                f'DATA DRIFT ALERT: {feature} PSI={psi:.3f} > 0.2. '
                f'Model retraining required immediately!'
            )
    return drift_report

def reduce_memory(df: pd.DataFrame) -> pd.DataFrame:
    '''Downcast dtypes to reduce RAM from ~1.8GB to ~600MB.'''
    for col in df.select_dtypes('float64').columns:
        df[col] = df[col].astype('float32')
    for col in df.select_dtypes('int64').columns:
        df[col] = df[col].astype('int32')
    return df
def drop_high_null_cols(df: pd.DataFrame,
                         threshold: float = 0.9) -> pd.DataFrame:
    '''Drop columns where > 90% values are missing.'''
    null_rates = df.isnull().mean()
    drop_cols  = null_rates[null_rates > threshold].index.tolist()
    print(f'Dropping {len(drop_cols)} columns with >{threshold:.0%} nulls')
    return df.drop(columns=drop_cols)
def engineer_card_features(df: pd.DataFrame) -> pd.DataFrame:
    '''Frequency-encode card identifiers as fraud risk proxies.'''
    for col in ['card1', 'card2', 'card3', 'card5']:
        freq = df[col].value_counts(normalize=True)
        df[f'{col}_freq'] = df[col].map(freq)
    # Cross-feature: card1 + addr1 identifies unique cards more precisely
    df['uid'] = df['card1'].astype(str) + '_' + df['addr1'].astype(str)
    uid_freq  = df['uid'].value_counts(normalize=True)
    df['uid_freq'] = df['uid'].map(uid_freq)
    return df
def engineer_time_features(df: pd.DataFrame) -> pd.DataFrame:
    '''TransactionDT is relative seconds — extract cyclical features.'''
    # Modulo to extract day-of-week and hour-of-day approximations
    df['hour']         = (df['TransactionDT'] // 3600) % 24
    df['day_of_week']  = (df['TransactionDT'] // (3600 * 24)) % 7
    df['is_weekend']   = (df['day_of_week'] >= 5).astype(int)
    df['is_night']     = ((df['hour'] >= 22) | (df['hour'] <= 6)).astype(int)
    return df

# model/graph_features.py
import networkx as nx
import pandas as pd
import numpy as np
def build_fraud_graph(df: pd.DataFrame) -> nx.Graph:
    '''
    Build bipartite graph: card nodes <-> device nodes
    Edge = a transaction occurred between card and device.
    High-degree device nodes = card testing rings.
    '''
    G = nx.Graph()
    for _, row in df.iterrows():
        card_node   = f"card_{row['card1']}"
        device_node = f"device_{row['DeviceInfo']}"
        G.add_node(card_node,   node_type='card')
        G.add_node(device_node, node_type='device')
        G.add_edge(card_node, device_node,
                   amount=row['TransactionAmt'],
                   is_fraud=row['isFraud'])
    return G
def extract_graph_features(df: pd.DataFrame,
                            G: nx.Graph) -> pd.DataFrame:
    '''
    For each transaction, compute graph-based risk signals.
    These become new columns in your feature matrix.
    '''
    graph_feats = []
    for _, row in df.iterrows():
        card_node   = f"card_{row['card1']}"
        device_node = f"device_{row['DeviceInfo']}"
        # How many cards share this device? (ring signal)
        device_degree = G.degree(device_node) if device_node in G else 0
        # How many devices has this card been used on? (takeover signal)
        card_degree   = G.degree(card_node) if card_node in G else 0
        # What % of device's connected cards are fraudulent?
        if device_node in G and G.degree(device_node) > 0:
            neighbors  = list(G.neighbors(device_node))
            fraud_nbrs = sum(
                1 for n in neighbors
                if any(G[device_node][n].get('is_fraud', 0)
                       for _ in [None])
            )
            device_fraud_rate = fraud_nbrs / len(neighbors)
        else:
            device_fraud_rate = 0.0
        graph_feats.append({
            'device_degree':      device_degree,
            'card_degree':        card_degree,
            'device_fraud_rate':  device_fraud_rate,
            'is_shared_device':   int(device_degree > 3),
            'is_card_ring_member': int(device_degree > 10),
        })
    return pd.DataFrame(graph_feats, index=df.index)

import xgboost as xgb
import numpy as np
import pandas as pd
import matplotlib.pyplot as plt
def compute_shap_values(model: xgb.XGBClassifier,
                         X: pd.DataFrame) -> shap.Explanation:
    '''Compute SHAP values using TreeExplainer (fast for XGBoost).'''
    explainer   = shap.TreeExplainer(model)
    shap_values = explainer(X)  # Returns Explanation object
    return shap_values
def plot_shap_summary(shap_values, X: pd.DataFrame,
                       output_path: str = 'reports/shap_summary.png'):
    '''Generate beeswarm plot — include this in your README.'''
    plt.figure(figsize=(10, 8))
    shap.summary_plot(shap_values, X, show=False)
    plt.tight_layout()
    plt.savefig(output_path, dpi=150, bbox_inches='tight')
    plt.close()
    print(f'SHAP summary plot saved to {output_path}')
def get_top_reasons(shap_values_row: np.ndarray,
                     feature_names: list,
                     n: int = 3) -> list[dict]:
    '''
    For a single prediction, return top-N features driving the score.
    Used by the /v1/explain API endpoint.
    '''
    pairs = list(zip(feature_names, shap_values_row))
    # Sort by absolute SHAP value — biggest impact first
    pairs.sort(key=lambda x: abs(x[1]), reverse=True)
    return [
        {
            'feature':   name,
            'shap_value': round(float(val), 4),
            'direction': 'increases_fraud_risk' if val > 0
                         else 'decreases_fraud_risk',
        }
        for name, val in pairs[:n]
    ]

import json, joblib, numpy as np
from sklearn.metrics import average_precision_score
import pandas as pd
def compare_models(candidate_path: str, baseline_path: str,
                    test_data_path: str, output_path: str):
    candidate = joblib.load(candidate_path)
    baseline  = joblib.load(baseline_path)
    df        = pd.read_parquet(test_data_path)
    from model.train import FEATURE_COLS, engineer_features
    df = engineer_features(df)
    X, y = df[FEATURE_COLS].fillna(0), df['isFraud']
    cand_auprc = average_precision_score(y, candidate.predict_proba(X)[:,1])
    base_auprc = average_precision_score(y, baseline.predict_proba(X)[:,1])
    result = {'candidate_auprc': cand_auprc, 'baseline_auprc': base_auprc,
              'delta': cand_auprc - base_auprc,
              'promote': cand_auprc >= base_auprc - 0.01}
    with open(output_path, 'w') as f:
        json.dump(result, f, indent=2)
    print(json.dumps(result, indent=2))
    return result

1. Gradient Boosting — What scale_pos_weight Actually Does
XGBoost minimizes a loss function via gradient descent in function space. For binary classification,
the default loss is log-loss (binary cross-entropy). The scale_pos_weight parameter multiplies the
gradient and hessian contributions of positive (fraud) samples by that factor. Here is what this means:
PYTHON
# MATHEMATICAL INTUITION (no code — read and understand this):
# Standard log-loss gradient for one sample:
#   g_i = pred_i - label_i   (first derivative)
#   h_i = pred_i * (1 - pred_i)  (second derivative / hessian)
# With scale_pos_weight=333 (for 0.3% fraud rate):
#   For fraud samples (label=1):  g_i *= 333,  h_i *= 333
#   For legit samples (label=0):  g_i *= 1,    h_i *= 1
# Effect: The model sees each fraud sample as if it appeared 333 times.
# This forces the tree splits to prioritize fraud separation.
# WHY AUPRC IS THE RIGHT METRIC (not AUC-ROC):
# AUC-ROC measures discrimination across ALL thresholds.
# At 0.3% fraud rate, a model that flags everything as fraud
# still gets AUC-ROC > 0.5 because it separates classes.
# AUPRC (Area Under Precision-Recall Curve) punishes the model
# for high false positive rate on the majority (legit) class.
# In fraud: FP = wrongly declined legit customer = lost revenue.
# AUPRC directly measures the trade-off you care about.
# INTERVIEW QUESTION: 'Why not just use accuracy?'
# ANSWER: 'With 0.3% fraud rate, a model predicting all-legitimate
# achieves 99.7% accuracy but catches zero fraud. Accuracy is
# meaningless for severely imbalanced problems. AUPRC at our
# operating threshold of 0.15 is the right business metric.'
2. PSI vs KL-Divergence — The Difference Interviewers Probe
Both PSI and KL-divergence measure distribution shift. Interviewers at senior levels will ask you to
distinguish them. Here is the answer:
Property

KL-Divergence
Formula
SUM((P-Q) * ln(P/Q))

Symmetric?
YES — PSI(P,Q) == PSI(Q,P)
NO — KL(P,Q) != KL(Q,P)
Interpretable thresholds?
YES — 0.1/0.2/0.25 industry standard
NO — no standard thresholds

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT user_id, COUNT(*) OVER (
    PARTITION BY user_id ORDER BY timestamp
    RANGE BETWEEN INTERVAL '10 minutes' PRECEDING AND CURRENT ROW
) AS txn_count_10m
FROM transactions
WHERE user_id = 'user_00001'
  AND timestamp >= NOW() - INTERVAL '1 hour';
-- WHAT TO LOOK FOR IN THE OUTPUT:
-- 1. 'Index Scan using idx_txn_user_time' -> GOOD
--    'Seq Scan on transactions' -> BAD (missing index, fix immediately)
-- 2. 'Rows Removed by Filter: 0' -> index is selective
--    'Rows Removed by Filter: 99000' -> index not helping, investigate
-- 3. 'WindowAgg' node -> this is the window function execution
--    cost=X..Y: X=startup cost, Y=total cost. Lower is better.
-- 4. 'Buffers: shared hit=N' -> N pages read from RAM cache (fast)
--    'Buffers: shared read=N' -> N pages read from disk (slow)
--    If 'read' >> 'hit': your working set doesn't fit in shared_buffers.
--    Fix: increase shared_buffers in postgresql.conf to 25% of RAM.
-- TARGET: For a 10M-row hypertable, your velocity query should:
--   Execute in < 10ms (index scan, not seq scan)
--   Show 'Buffers: shared hit' >> 'shared read' (hot cache)
